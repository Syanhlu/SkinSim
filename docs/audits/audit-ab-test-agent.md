## 1. FEATURE INVENTORY

Paths are relative to `vng-ab-test-agent/`. This was a read-only source audit; I did not run build/eval, and `node_modules` is absent in the checkout.

- **npm scripts:** `dev`, `build`, `start`, `lint`, and `eval` exist in `package.json:5-10`. There is no `test` script.
- **`/` home page:** `app/page.tsx` is a client component (`app/page.tsx:1`). It renders a hypothesis textarea, canned scenario buttons, an instant test brief, an instant readout report, a JSON tool trace, and an AI chat stream (`app/page.tsx:39-164`). State: **mock-only UI**, source-level plausible, not a real async experiment launcher.
- **Inline UI components:** `Metric`, `Row`, and `DecisionBadge` are defined locally, not imported from a components directory (`app/page.tsx:171-190`). `app/page.tsx` imports no external component folders (`app/page.tsx:3-8`).

It defines deterministic tools: `parse_hypothesis`, `power_analysis`, `design_test`, `significance_test`, `check_guardrails`, and `recommend` (`app/api/agent/route.ts:93-177`). - **Docs:** `README.md` describes the older stats-agent story and MiroShark as demo-data generator (`README.md:39-68`), then claims post-audit fixes are resolved (`README.md:131-163`). `SCAFFOLD.md` still mentions Supabase/STATS_URL and old Python API files (`SCAFFOLD.md:14-26`). ## 3. PROD-READINESS ISSUES

- **Real MiroShark integration will not satisfy the planned API contract.** The plan says POST returns a job and results come from a later GET (`../VNG_GRAND_PLAN.md:127-157`), but `MiroSharkClient` returns the POST body as `ExperimentResults` immediately (`lib/sim-client.ts:21-29`). It also omits the required internal key header (`../VNG_GRAND_PLAN.md:213-214`, `lib/sim-client.ts:22-25`).
- **No fallback if MiroShark is configured but unavailable.** Grand plan requires demo survival with MiroShark down (`../VNG_GRAND_PLAN.md:31-33`). Current fallback only happens when `MIROSHARK_URL` is unset (`lib/sim-client.ts:32-35`); fetch failure or non-OK throws (`lib/sim-client.ts:27`).
- **API request validation is thin.** `/api/agent` trusts `await req.json()` as `{ messages: UIMessage[] }` with no zod validation or error response (`app/api/agent/route.ts:85-91`). Bad payloads become server errors.
- **Runtime result validation is missing.** The route defines `resultsSchema` for tool-supplied inputs (`app/api/agent/route.ts:36-83`), but external MiroShark JSON is cast directly (`lib/sim-client.ts:28`) and then cast again into stats calls (`app/api/agent/route.ts:142-155`, `167-175`).
- **Env handling is incomplete.** `.env.example` documents `MIROSHARK_URL` but not `MIROSHARK_INTERNAL_KEY` (`.env.example:8-17`), while the plan requires that key (`../VNG_GRAND_PLAN.md:411-413`). The model is hardcoded in code (`lib/ai.ts:8-9`), and the UI always shows “Ask agent” even if no AI key is configured (`app/page.tsx:63-65`).
- **Client key exposure risk appears low, but server auth is missing.** The client page imports local mock/stats modules only (`app/page.tsx:3-8`); server-only model/sim imports are in the route (`app/api/agent/route.ts:3-12`). I found no client import of secrets. The concrete security gap is the missing MiroShark internal auth header (`lib/sim-client.ts:22-25`).
- **Hypothesis parsing silently guesses.** Empty or weird input falls through to “Purchase conversion” with baseline `0.06` and MDE `0.01` (`lib/experiment.ts:120-129`). There is no UI validation around the textarea (`app/page.tsx:41-47`).
- **Hardcoded assumptions dominate the product.** Baselines/MDEs are fixed by keywords (`lib/experiment.ts:80-129`), daily traffic defaults to `1600` in both core and agent tools (`lib/experiment.ts:132`, `app/api/agent/route.ts:105-113`), and guardrails are fixed strings (`lib/experiment.ts:158-165`).
- **Async state can drift.** The readout recomputes from current `hypothesis` and `scenario` (`app/page.tsx:13-15`), while `sendToAgent` sends a snapshot string (`app/page.tsx:20-23`). Users can still edit textarea/scenario buttons during an agent run (`app/page.tsx:42-55`); only the Ask button is disabled (`app/page.tsx:63-65`).
- **Loading and error feedback is minimal.** Agent status is rendered as raw text (`app/page.tsx:131-134`), empty state is explanatory copy only (`app/page.tsx:136-141`), disabled button styling is just cursor/opacity (`app/page.tsx:382-385`), and no error object/toast/inline error is rendered from `useChat`.
- **Accessibility gaps:** scenario selection uses visual `active` class only, no `aria-pressed`/`aria-selected` (`app/page.tsx:49-59`, `368-371`); streamed messages are not an aria-live region (`app/page.tsx:135-160`); focus-visible styles are not defined for buttons or textarea in the CSS block (`app/page.tsx:343-380`).
- **Live-demo embarrassment risk:** the UI copy promises “run deterministic MiroShark mock data” (`app/page.tsx:33-35`), but the “run” is just instantaneous local recomputation (`app/page.tsx:15`) and the only action is “Ask agent” (`app/page.tsx:63-65`). There is no launch, progress, same-audience visualization, or real job status.

## 4. UX AUDIT

- On load, the user sees a polished single-page dashboard: eyebrow, large headline, and note explaining hypothesis to powered brief to mock stats readout (`app/page.tsx:28-36`).
- The first workflow is a textarea prefilled with “A red Buy button…” (`app/page.tsx:10-14`, `41-47`), followed by six scenario buttons from `demoScenarios` (`app/page.tsx:49-60`, `lib/mock-results.ts:15-46`). Friction: these are canned outcomes, but the UI does not visibly label them as outcome presets except the hidden aria label.
- The test brief updates immediately as the user types or changes scenario (`app/page.tsx:15`, `68-85`). Friction: metric, baseline, MDE, sample size, duration, variants, stop rule, and guardrails are read-only display rows, not confirmable/editable operator inputs.
- The readout report appears immediately beside it with decision badge, p-value, effect, CI, sample, guardrail state, and caveats (`app/page.tsx:87-111`). Friction: it feels like a completed experiment before the user has launched anything.
- The lower-left “Tool Trace” shows raw JSON in expandable `<details>` blocks (`app/page.tsx:114-128`). This is credible for a judge, but heavy for a product user, and the dark code block visually competes with the decision.
- The lower-right “Agent Stream” is separate from the readout (`app/page.tsx:130-164`). Clicking Ask agent sends the current hypothesis/scenario to `/api/agent` (`app/page.tsx:20-23`), but returned tool calls do not update the cards. Dead end: after the stream, the user has no next action.
- The product story from the grand plan is not represented: no editable brief confirmation, no variant cards, no launch/progress, no split-world audience, no interview, no replay fallback, no per-region chips. The current UI is a classic stats credibility card, not the “agent world” product face.

## 5. TOP 10 CONCRETE FEATURE SUGGESTIONS

1. **Implement the Phase 3 job-based sim proxy.** Add `app/api/experiment/route.ts`, `createExperiment/getStatus/getResults`, internal key header, and mock polling. This fixes the broken real-client contract in `lib/sim-client.ts:21-29`. Effort: **M/L**.
2. **Add structured hypothesis extraction plus editable confirmation.** Replace keyword-only `parseHypothesis` (`lib/experiment.ts:77-130`) with `extractHypothesis` and show editable metric, baseline, MDE, unit, direction before launch. Effort: **M**.
3. **Separate “preview” from “run.”** Keep canned scenarios for demos, but add a real Launch state with preparing/running/complete/error so the readout does not appear as magic recomputation from `app/page.tsx:15`. Effort: **M**.
4. **Build a minimal `/world` replay first.** Use bundled JSON and a split-screen same-personas layout before live polling. This addresses the complete Phase 4 absence while keeping stage safety. Effort: **L**.
5. **Create variant cards before launch.** Add `proposeVariants` and editable Vietnamese copy cards so a single hypothesis can become two launchable treatments, matching Phase 5. Effort: **M**.
6. **Harden errors and fallbacks.** Show missing AI key, MiroShark down, bad payload, and failed stats states inline; fallback to mock when real sim fails. Current route and client throw without UX recovery (`app/api/agent/route.ts:85-91`, `lib/sim-client.ts:27`). Effort: **S/M**.
7. **Validate all external results at runtime.** Reuse the existing zod result schema or move it to `lib/` so `res.json()` is parsed, not cast (`lib/sim-client.ts:28`). Effort: **S**.
8. **Fix the scenario selector and stream accessibility.** Add visible “Demo outcome preset” label, `aria-pressed`, keyboard/focus styles, and `aria-live` for agent messages (`app/page.tsx:49-59`, `135-160`). Effort: **S**.
9. **Extract reusable readout components.** Move `Metric`, `Row`, `DecisionBadge`, and readout card rendering out of `app/page.tsx:171-190` so `/classic` and `/world` can share one stats presentation. Effort: **M**.
10. **Upgrade eval to cover integration contracts.** Keep the deterministic stats eval, but add tests for Phase 3 client/proxy shape and a mocked `/api/agent` tool path. Current agent consistency is useful but only reconstructs tools locally over `MockSimClient` (`eval/agent-consistency.ts:29-57`). Effort: **M**.
