**2. A/B Experiment Harness Audit (ab_experiment.py)**
- Behavior: CLI args cover API URL, internal key, parent/scenario input, variants, replicates, parallel cap, trigger round, output dir, dry-run, resume (`MiroShark/backend/scripts/ab_experiment.py:223-260`). It validates parent-vs-scenario, requires at least two variants, and blocks duplicate names (`MiroShark/backend/scripts/ab_experiment.py:264-291`).
- Parent setup: creates ontology, builds graph, creates simulation, then prepares the parent (`MiroShark/backend/scripts/ab_experiment.py:463-536`); existing parents are prepared via `/api/simulation/prepare` and status polling (`MiroShark/backend/scripts/ab_experiment.py:415-460`).
- Branch/run loop: for each variant replicate it calls `/api/simulation/branch-counterfactual`, then `/api/simulation/start`; branch/start failures mark that run failed and persist state (`MiroShark/backend/scripts/ab_experiment.py:546-606`).
- Parallel cap: it only starts new runs while `len(active_runs) < parallel`, then polls active runs (`MiroShark/backend/scripts/ab_experiment.py:694-716`).
- Resume: `--resume` requires `--out`, loads `<out>/state.json`, and skips runs already marked `metrics_fetched` (`MiroShark/backend/scripts/ab_experiment.py:274-277`, `MiroShark/backend/scripts/ab_experiment.py:1192-1201`, `MiroShark/backend/scripts/ab_experiment.py:749-756`). It does **not** pass `resume=true` to `/api/simulation/start`; the start body only includes `simulation_id` (`MiroShark/backend/scripts/ab_experiment.py:587-592`) while the API supports resume (`MiroShark/backend/app/api/simulation.py:3192-3200`).
- HTTP robustness: `ApiClient` retries each request up to three times, attaches `x-miroshark-internal-key` when configured, parses JSON, and returns error envelopes instead of throwing from `_request` (`MiroShark/backend/scripts/ab_experiment.py:176-212`).
- Failure handling: parent/graph/prepare failures raise and abort (`MiroShark/backend/scripts/ab_experiment.py:390-403`, `MiroShark/backend/scripts/ab_experiment.py:483-530`); run-level branch/start failures are recorded and the matrix continues (`MiroShark/backend/scripts/ab_experiment.py:568-600`).
- Metrics: final bullish % comes from `/belief-drift`; demographics and cost are fetched and stored even if unavailable (`MiroShark/backend/scripts/ab_experiment.py:719-737`, `MiroShark/backend/scripts/ab_experiment.py:749-784`).
- Statistics: per-variant mean/std/95% CI use replicate final bullish percentages; pairwise comparison is Welch vs the first variant, with “need at least two observations” guard (`MiroShark/backend/scripts/ab_experiment.py:801-842`, `MiroShark/backend/scripts/ab_experiment.py:851-897`).
- Report output: includes parent IDs, variant summary, demographic winners, per-run cost table, total available cost, and endpoints used (`MiroShark/backend/scripts/ab_experiment.py:1043-1153`, `MiroShark/backend/scripts/ab_experiment.py:1156-1172`).

**6. Production-Readiness & Demo-Risk Findings**
1. **Critical: Real Next.js experiment flow cannot hit MiroShark yet.** Missing `/api/experiments/*` in MiroShark, while Next already calls `/api/experiments/ab-test` without internal key (`MiroShark/backend/app/__init__.py:166-230`, `vng-ab-test-agent/lib/sim-client.ts:21-28`).
2. **Critical: Demo may still show non-discriminating variants.** Status doc says prior run was 100% bullish and requires richer scenario + VN census (`MiroShark/docs/VNG_AB_STATUS.md:51-60`); scenario file is not found and VN env is commented (`MiroShark/.env:47-48`).
3. **High: Live run time/cost exposure.** Plan says full sim is ~$1-2 and 10-20 min, validation ~$3-5 and ~30 min (`VNG_GRAND_PLAN.md:35`, `VNG_GRAND_PLAN.md:76`); Wonderwall is documented as 850+ calls/run (`MiroShark/backend/app/config.py:180-184`).
4. **High: Report cost likely remains unavailable for private experiment branches.** Harness fetches `cost.json` (`MiroShark/backend/scripts/ab_experiment.py:768-770`), but endpoint rejects unpublished sims (`MiroShark/backend/app/api/simulation.py:6020-6028`).
5. **High: Runner state is process-local and subprocess-based.** Run/process/thread registries are in-memory (`MiroShark/backend/app/services/simulation_runner.py:103-107`); simulations spawn subprocesses plus daemon monitor threads (`MiroShark/backend/app/services/simulation_runner.py:431-459`). Phase 2’s planned single-worker background-thread constraint is not implemented (`VNG_GRAND_PLAN.md:112-118`).
6. **Medium: TikTok is not in the synchronized A/B runner.** It is standalone only (`MiroShark/backend/app/services/simulation_manager.py:68-71`, `MiroShark/backend/app/services/simulation_manager.py:813-825`).
7. **Medium: Windows/setup friction.** Status doc requires short venv path and venv python (`MiroShark/docs/VNG_AB_STATUS.md:79-88`); `run.py` has Windows UTF-8 fixes (`MiroShark/backend/run.py:8-16`); root npm `predev` uses Unix `lsof/kill` (`MiroShark/package.json:9`).
8. **Medium: Dependency path mismatch.** `requirements.txt` includes `duckdb`/`huggingface_hub` but not `sentence-transformers` (`MiroShark/backend/requirements.txt:50-56`); `pyproject.toml` includes `sentence-transformers` and `torch` (`MiroShark/backend/pyproject.toml:33-40`).
9. **Medium: Deployment docs drift.** `README_DEPLOYMENT.md` says backend does not prefer Railway `PORT` and auth guard is missing (`MiroShark/README_DEPLOYMENT.md:23-28`, `MiroShark/README_DEPLOYMENT.md:47-54`), but current code does prefer `PORT` and has the guard (`MiroShark/backend/run.py:40-46`, `MiroShark/backend/app/__init__.py:91-158`).
